# 个人微信视觉 AI 助手 (GUI Agent) 开发指南

**核心目的与破局思路**
本项目旨在从零开发一款完全脱离底层协议、100% 免疫微信封号风控的"纯视觉"个人微信 AI 助手，并为未来的开源发布奠定标准化的工程基础。我们彻底摒弃了高危的内存 Hook 与 API 逆向，转而采用"GUI 智能体"架构：让程序像真实人类一样，通过将窗口拉至前台、截屏"看"消息、利用 OCR"读"文字、结合本地知识库与大模型大脑进行"思考"，最终通过物理模拟键鼠完成"回复"。这不仅是一套极低成本、极高隐私的自动化客服底座，更是全栈开发向多模态 AI Agent 进阶的绝佳实践。

---

## 一、 核心架构与工作流设计

为了兼顾性能、成本和成功率，系统将采用**分层监听与读写分离**的设计。每次循环必须严格遵循以下流水线：

1. **窗口激活与前台锁定 (Focus & Foreground)**
    - **动作**：在进行任何视觉捕获前，通过操作系统 API 获取微信窗口句柄，强制将其从后台唤醒、取消最小化，并置于屏幕最顶层。
    - **意义**：彻底解决窗口被其他软件遮挡、截图出现黑屏或截到其他内容导致的 OCR 彻底瘫痪问题，保证视觉输入的 100% 纯净度。
2. **状态轮询与轻量级监听 (Visual Polling)**
    - **动作**：**弃用消耗极高且极易误判的全区域 SSIM 对比**。对激活后的微信左侧会话列表进行极小特定区域截图。将其转为 HSV 色彩空间，仅寻找和匹配“未读消息红点”的特定红色像素，或通过提取红点旁最新消息的缩略文本块进行比对。
    - **意义**：避免由于群聊时间戳更新、打字状态闪动等引起的无效触发，极大降低 CPU 运算负荷，实现真正的"零 Token 消耗"安静待机。
    - **首次校准机制**：由于不同用户的屏幕分辨率、系统缩放比例和微信窗口大小各不相同，程序**首次启动时**需引导用户完成一次**交互式区域标定**：
        1. 程序弹窗提示"请用鼠标框选微信**左侧会话列表**区域"，用户拖拽后记录坐标 `session_list_rect`；
        2. 程序弹窗提示"请用鼠标框选微信**右侧聊天内容**区域"，用户拖拽后记录坐标 `chat_content_rect`；
        3. 程序弹窗提示"请用鼠标点击微信**底部输入框**中心位置"，用户点击后记录坐标 `input_box_center`；
        4. 将以上坐标作为相对于微信窗口左上角的**偏移量**保存到 `config.yaml`，后续运行通过窗口句柄实时获取窗口位置并换算绝对坐标。
    - **系统 DPI 缩放适配（核心避坑）**：由于 Windows 系统普遍存在 125%、150% 显示缩放，`pyautogui` 模拟鼠标的逻辑坐标将与 `mss` / `OpenCV` 截屏抓取的物理像素坐标产生严重偏差。**程序启动时必须强制调用 Windows DPI 感知 API** (`ctypes.windll.shcore.SetProcessDpiAwareness(2)`)，确保所有物理坐标系完全统一。
    - **浅色模式依赖要求**：MVP 阶段，**强制要求微信客户端保持在「浅色模式」**运行。视觉找图高度依赖气泡颜色和红点像素，开启深色模式（Dark Mode）会导致所有硬编码图像阈值失效。
    - **分辨率自适应**：如果微信窗口大小发生变化（用户手动调整或切换显示器），程序应检测到窗口尺寸变动后提示用户重新校准，或按比例自动缩放已保存坐标。
3. **语义提取 (OCR Parsing)**
    - **动作**：一旦检测到新消息提示（如绿色气泡或红点），截取聊天记录区域，送入本地轻量级 OCR 引擎/图像识别大模型提取纯文本及对应屏幕物理坐标。
    - **消息归属判断规则（核心难点）**：OCR 返回的是一堆"文字块 + 坐标"，但程序必须知道**哪句话是对方说的、哪句话是自己说的**。微信聊天界面的视觉布局有严格规律：
        - **对方消息**：气泡/文本块的 X 坐标偏向聊天区域**左侧**（`x < chat_width * 40%`）
        - **自己的消息**：气泡/文本块的 X 坐标偏向聊天区域**右侧**（`x > chat_width * 60%`）
        - **系统时间戳/通知**：文本块位于聊天区域**水平居中**位置（`40% <= x <= 60%`），字体通常为灰色小号，应直接**过滤忽略**
        - **多模态异常处理与兜底策略（核心机制）**：当 OCR 提取出 `[语音]`、`[图片]`、`[视频]`、`[文件]` 等特定系统提示文本，或遇到小程序/红包等无法解析的卡片元素时，说明接收到了暂不支持的多模态消息。此时**触发专门的兜底降级响应**，不送入大模型，直接由 Agent 执行固定人设回复话术：**“抱歉，我现在的设备听不了语音/看不了图片，麻烦发下文字哦~”**，以避免话题阻断或无反应卡顿。
    - **消息去重与状态管理（禁止依赖 Y 坐标）**：OCR 每次截图都会识别出屏幕上**所有可见消息**，但程序只应回复新增的、未处理过的消息。由于用户可能滚动鼠标，甚至后续新消息涌入也会顶替屏幕布局，任何带绝对坐标的去重机制都会引发“无限重复回复”的致命灾难。正确设计如下：
        1. 废弃对绝对 Y 坐标的依赖。改为对每条识别到的消息，计算纯内容哈希值（`MD5(消息文本内容 + 发送方标识)`）。
        2. 为了应对用户连续发送两条完全一模一样的短文本（导致哈希无法区分），引入**滑动窗口序列缓存**机制：不仅对比单条消息特征，还要将“最近 N 条连续消息的内容列表”作为当前对话进度的指纹进行保存与对比。
        3. 维护一个本地内存的 `已处理消息序列集`，以此确保绝对且唯一的消息消费。
        5. 集合大小超过阈值（如 500 条）时，移除最早的条目，防止内存无限增长
4. **大脑决策 (LLM + RAG)**
    - **动作**：将提取到的文本，结合预设的系统人设（Persona）以及本地向量数据库检索到的专业知识，组装成 Prompt 发送给大语言模型。
5. **拟人化执行 (Human-like Action)**
    - **动作**：计算输入框的绝对坐标，通过贝塞尔曲线移动鼠标点击激活，加入随机思考延迟后，将 AI 回复通过剪贴板安全注入并发送。
    - **详见下方"第四章：深度拟人化引擎"**。

## 二、 推荐技术栈 (向开源与跨平台对齐)

构建这套系统，建议采用前后端解耦的模式，方便后期打包开源。

- **核心控制大脑 (后端)**：`Python 3.10+`
    - **窗口与键鼠控制**：`pygetwindow` (跨平台窗口管理) + `pyautogui` / `pynput` (模拟操作)。
    - **视觉与图像**：`OpenCV` (`cv2`) 负责高效找图和对比，`mss` 负责极低延迟的屏幕截图。
    - **文本识别**：`PaddleOCR` (百度开源，对中文极度友好，本地运行极快)。
- **知识库与大模型对接**：
    - **RAG 框架**：`LangChain` 配合 `ChromaDB` (轻量级本地向量库)。
    - **LLM 接口**：采用兼容 OpenAI 格式的封装。这样既可以无缝对接云端 API 跑通 MVP，也能随时切回你本地部署的开源大模型（比如在独立的推理节点上跑），把隐私彻底锁在本地。
- **可视化控制台 (前端/未来规划)**：
    - 后期如果想加个漂亮的悬浮控制台来展示"思考过程"和管理知识库，直接上 `React` + `TypeScript` 配合 `Electron` 封装，这是全栈最顺手的打法，打包给开源社区也极其友好。

## 三、 分阶段开发路径

为控制工程复杂度，项目开发严格分为两个阶段。**先跑通 MVP 闭环，再逐步加载进阶能力。**

### MVP 阶段（核心目标：端到端闭环跑通）

| 步骤 | 任务 | 交付物 |
|------|------|--------|
| M1 | 环境初始化：安装 Python 依赖，创建项目骨架文件和 `config.yaml` | `requirements.txt`、项目目录结构 |
| M2 | 窗口控制模块：实现微信窗口的发现、激活、前台锁定 | `window_manager.py` |
| M3 | 截屏与视觉差异检测：实现区域截屏、首次校准交互、SSIM 差异对比 | `vision.py` |
| M4 | OCR 提取与消息解析：集成 PaddleOCR，实现消息文本提取 + 归属判断 + 去重 | `ocr_parser.py` |
| M5 | LLM 直连（不上 RAG）：封装 OpenAI 兼容接口，设计基础人设 Prompt | `agent.py` |
| M6 | 执行层：实现剪贴板粘贴 + 物理键盘发送的基础回复链路 | `action.py` |
| M7 | 主循环串联：将 M2~M6 串入 `main.py`，完成一问一答闭环 | `main.py` |
| M8 | 基础拟人化：加入思考延迟（泊松分布）和作息时间墙 | `anti_risk.py` |

### 进阶阶段（核心目标：商业可用 + 开源就绪）

| 步骤 | 任务 | 交付物 |
|------|------|--------|
| A1 | RAG 知识库接入：集成 LangChain + ChromaDB，支持本地业务资料检索 | `rag_engine.py` |
| A2 | 深度拟人化：贝塞尔曲线鼠标轨迹、手误模拟、非匀速击键 | `anti_risk.py` 增强 |
| A3 | 多会话管理：支持自动切换左侧会话列表的不同联系人 | `session_manager.py` |
| A4 | Electron 可视化控制台：展示思考过程、管理知识库、一键启停 | `console/` 前端项目 |
| A5 | 开源打包：README、LICENSE、Docker 化部署、GitHub Actions CI | 项目根目录 |

## 四、 深度拟人化引擎 (Anti-Risk Engine)

纯粹的自动化极易被云端行为特征库识别。必须在代码逻辑中注入"人类的瑕疵"，让程序的每一个动作都带有随机性和生理特征。

### 4.1 泊松分布的"思考延迟" (Thinking Delay)
人类回复消息的耗时与消息长度呈正相关，且具备随机性。收到消息后，程序必须进入挂起状态。
- **公式参考**：`基础延迟 T = (接收消息字数 * 0.2秒) + random(1, 3)秒`
- 只有等待期结束，才允许向大模型发起请求。大模型返回后，再追加 `random(0.5, 1.5)秒` 的"打字前犹豫"。

### 4.2 仿生作息时间墙 (Biological Clock)
阻断 24 小时高频秒回的机器特征。
- **强制休眠期**：每日 `00:00 - 07:00` 不进行任何回复。
- **慢速时段**：午休 `12:00 - 13:30` 和晚餐 `18:00 - 19:30`，将整体回复延迟基数乘以 `1.5 ~ 2.0`，模拟"正在忙碌，偶尔看一眼电脑"的真实状态。
- **深夜降速**：`22:00 - 00:00` 回复延迟基数乘以 `1.3`，模拟困意渐浓。

### 4.3 非匀速与手误模拟 (Variable Keystrokes & Typos)
- 在使用物理模拟敲击的场景中，每次按键间隔必须在 `0.08秒 ~ 0.25秒` 之间随机波动。
- **高级伪装**：设定 `2%` 的全局手误率。偶尔故意输入错误的相邻字母，停顿 `0.3 ~ 0.8秒` 后，模拟敲击 `Backspace` 键删除，再输入正确字母。
- **注意**：在 MVP 阶段我们采用剪贴板粘贴方案（`Ctrl+V`），手误模拟为进阶阶段功能。

### 4.4 鼠标轨迹的贝塞尔曲线 (Mouse Trajectory)
- **禁止瞬移**：鼠标指针在屏幕上必须走贝塞尔曲线路径，加入弧度和接近目标时的减速（Ease-Out）效果。
- **微抖动**：到达目标坐标后，在点击前加入 `+/-2~5像素` 的微小偏移抖动，模拟人手的不精确性。
- **进阶阶段实现**：MVP 阶段可先用 `pyautogui.moveTo()` 的内置 `duration` 参数做简单过渡。

## 五、 开发要点与避坑指南

### 1. 窗口前台激活的"系统级玄学"

在 Windows 系统下，频繁抢夺焦点极易触发系统的"防骚扰机制"（窗口只在任务栏闪烁而不弹到前台）。

- **避坑对策**：需要调用 `ctypes.windll.user32` 底层接口，在调用 `SetForegroundWindow` 之前，模拟一次按下 `Alt` 键的动作，可以100% 绕过 Windows 的限制，强制拉起微信。

### 2. OCR 性能优化：不要"大水漫灌"

- **避坑对策**：绝对不要把 1080P 或 4K 的全屏截图直接扔给 OCR。必须先用 OpenCV 裁剪出只包含"最新一条聊天气泡"的极小矩形区域（可能只有 300x100 像素），再进行 OCR 识别。这能把识别时间从 1-2 秒压缩到 50 毫秒以内。

### 3. 垂直领域的知识挂载 (RAG)

如果要让这个助手具备商业价值，知识库是灵魂。

- **开发思路**：把你的业务资料存入向量数据库。比如针对电商场景，把实体商品（如水晶、首饰）的材质说明、白皮书，或者数字服务（如 AI 测算报告）的价目表切片存入。当客户询问"这个手链是什么材质"时，大模型会优先从本地检索事实，避免胡编乱造。

### 4. 开源架构设计 (Open-Source Readiness)

目标是开源，第一天就要写好抽象层：

- 配置文件抽离：使用 `.env` 或 `config.yaml` 管理屏幕分辨率、OCR 引擎路径、API Key 等，绝对不要硬编码。
- 模块化设计：将 `Vision` (看)、`Agent` (想)、`Action` (做) 拆分成独立的 Python 类，方便社区后续贡献新的视觉模型或对接不同的聊天软件。

### 5. 异常与容错处理 (Fault Tolerance)

视觉方案最脆弱的地方在于截屏和 OCR 随时可能失败，必须构建多层防护网：

- **OCR 置信度过滤**：PaddleOCR 返回的每条文本都附带置信度分数（0~1）。当置信度低于阈值（建议 `0.7`）时，该条文字应**丢弃或标记为不可信**，绝不允许直接送入 LLM 产生幻觉回复。
- **窗口状态守护**：每次主循环开始前，必须校验微信窗口是否仍然存在（句柄有效性检查）。如果微信被用户手动关闭或崩溃退出，程序应进入**优雅降级模式**——暂停主循环并每隔 30 秒轮询微信进程是否重新启动，而不是直接抛异常崩溃。
- **焦点窗口二次验证（防错发保护）**：在执行 `Ctrl+V` 粘贴和 `Enter` 发送之前的**最后一刻**，必须通过 `GetForegroundWindow()` 再次确认当前最顶层窗口的标题/句柄仍然是微信。如果焦点已被其他弹窗抢占，则**立即中止本次操作**。
- **物理输入侦听锁（竞态条件防御）**：如果 Agent 正在执行“激活窗口 -> 点击 -> 粘贴 -> 发送”动作组时，真人用户刚好移动了鼠标或打字，将导致毁灭性的执行脱位。必须通过 `pynput` 等库侦听全局鼠标与键盘事件，当发现过去 3~5 秒内存在真实的物理输入，Agent 应判定当前处于“人类接管状态”，临时挂起当前的自动恢复流，直到物理输入停止后再重新获取焦点工作。
- **剪贴板竞争保护**：在写入剪贴板前，先备份当前剪贴板内容；写入完毕并确认发送后，立刻恢复原始剪贴板内容。
- **全局异常兜底**：主循环包裹在 `try-except` 中，任何未预期的异常被捕获并写入日志，程序在等待冷却期后自动尝试恢复运行，非毁灭性错误绝不直接退出。
